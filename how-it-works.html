Page Spotlight Speed Reader

How the extension works

A tour of the five core files behind this Manifest V3 Chrome extension — plain JavaScript, no build step, no frameworks.

The three worlds System map manifest.json background.js popup content.js A live run Repo layout

The five core source files live in three isolated execution contexts that can only talk to each other by passing messages. Understanding that separation is the key to understanding everything else.

Architecture

The three worlds

Chrome deliberately sandboxes extension code. A background worker can't touch a page's DOM; a popup can't run forever; a content script can't reach the popup's buttons. The files split responsibilities accordingly.

manifest.json the wiring diagram — declares permissions, worker, popup, and shortcuts. Runs no logic.
↓ wires together
service worker background.js Invisible and event-driven. Wakes on a keyboard shortcut, finds the active tab, ensures the content script is loaded, forwards a command. Owns no UI.
popup popup.html + popup.js The settings panel behind the toolbar icon: WPM, chunk size, auto-scroll, status, time estimate. Opens and closes with each click — not persistent.
content script content.js The muscle. Runs inside the web page: finds the article, wraps every word in a span, highlights one chunk at a time on a timer. Lets you start from a selected word or click a word to jump while reading. The only file that touches the page's DOM.
background ⇄ popup ⇄ content: SPEED_READER_* messages only

01 · Config

manifest.json — the wiring diagram

Everything starts here. Chrome reads this before any code runs: 

"permissions": ["activeTab", "scripting", "storage", "tabs"],
"host_permissions": ["<all_urls>"],
"background": { "service_worker": "background.js" },
"action": { "default_popup": "popup.html" },
"commands": {
  "toggle-reader": { "suggested_key": { "default": "Alt+Y", "mac": "Command+Shift+Y" } },
  "increase-speed": { ... }, "decrease-speed": { ... }, "stop-reader": { ... }
}

Two permissions matter most: scripting lets the extension inject content.js on demand (it's deliberately not listed under content_scripts), and storage persists settings across sessions.

Why lazy injection? A static content script would load into every page automatically, costing memory and risking breakage. Instead, content.js loads only when the user actually invokes the reader on a tab.
02 · Service worker

background.js — the keyboard dispatcher

The worker's only entry point is a single listener: 

chrome.commands.onCommand.addListener(async (command) => {
  const [tab] = await chrome.tabs.query({ active: true, currentWindow: true });
  if (!tab?.id || !isSupportedUrl(tab.url)) return;
  await ensureContentScript(tab.id);
  // ...then send the right SPEED_READER_* message
});

Every shortcut follows the same flow: find the active tab (bail if not http/https), run the ping-then-inject pattern, then send content.js its command. 

async function ensureContentScript(tabId) {
  try {
    await chrome.tabs.sendMessage(tabId, { type: "SPEED_READER_PING" });
  } catch {
    await chrome.scripting.executeScript({ target: { tabId }, files: ["content.js"] });
  }
}

If the ping succeeds, content.js is already there; if it throws, inject it. This makes shortcuts idempotent — on a fresh tab it loads + starts, on a running tab it just toggles. Speed up/down also writes to chrome.storage.sync, which is how a shortcut-pressed speed change appears live in an open popup.

03 · Popup

popup.html + popup.js — the control panel

The HTML is plain markup — status text, Start/Stop/Reset, a WPM stepper, chunk size, auto-scroll. On open, init() runs: 

const settings = await chrome.storage.sync.get(DEFAULT_SETTINGS);
applySettingsToInputs(settings);
const [tab] = await chrome.tabs.query({ active: true, currentWindow: true });
if (!state.supported) { renderStatus("unsupported"); return; }
await ensureContentScript();
const response = await sendMessage({ type: "SPEED_READER_STATUS" });
renderStatus(response?.status ?? "ready");
await refreshEstimate();

It reuses the same ping-then-inject pattern, then asks content.js for status and a word count so it can show "Article words / Est. time" immediately. Buttons send SPEED_READER_* messages and render the { status, estimate } reply. Crucially, it also listens for storage changes — so keyboard-shortcut speed changes handled by the worker update an open popup instantly.

04 · Content script

content.js — the reading engine

Where the actual work happens. A single IIFE guards against double-injection with a window flag, since Chrome can inject the file more than once: 

if (win.__pageSpotlightSpeedReader) return;
// ...set up everything...
win.__pageSpotlightSpeedReader = true;

The message handler

All commands arrive at one listener; every response has the same { status, estimate? } shape so the popup can render any reply generically.

MessageWhat content.js does
SPEED_READER_PING Returns status only — used to detect whether the script is loaded.
SPEED_READER_STATUS / _ESTIMATE Returns current status + word count, without changing anything.
SPEED_READER_TOGGLE Applies settings, then starts / pauses / resumes depending on current state.
SPEED_READER_STOP Stops and tears down, restoring the original DOM.
SPEED_READER_RESET Restores the page to its original state.
SPEED_READER_SETTINGS Updates WPM / chunk / scroll; if running, re-highlights and reschedules.

Finding the article

findReadableRoot() scores candidate elements (article, main, section, div) by visible text length plus 250 points per substantial <p>, with a heading bonus. It prefers a direct article/main scoring ≥ 120, else the highest-scoring candidate ≥ 240. If the chosen root covers under 85% of the page's readable words, it expands to the common ancestor of all readable blocks. Excluded selectors (nav, header, footer, ads, forms) and distraction heuristics (classes matching ad, newsletter, subscribe…) filter out junk.

Wrapping words

wrapWords() walks the readable blocks (p, li, blockquote, h1–h3, plus leaf divs on pages without real paragraphs), splits each text node into words, and replaces the text with <span class="pssr-word"> elements. Each original text node is saved in reader.originals so reset() can swap it back and leave the page exactly as it found it.

The same pass is where "start from anywhere" is resolved. Because wrapping replaces each text node, any selection pointing into it would be destroyed — so start() grabs the visible selection's start first, and wrapWords(root, anchor) records the first word span that extends past that caret as the starting index. Only a non-collapsed (highlighted) selection counts; a stale collapsed caret starts at the top. Once the reader is active, a single delegated click listener calls jumpTo(index) to move the position to any clicked word.

Pacing

scheduleNext() chains setTimeout calls at an interval of (60000 / wpm) × chunkSize, floored at 80 ms so very high WPM can't freeze the page. Each tick advances by chunkSize, highlights the current chunk, dims everything read, optionally smooth-scrolls the active word into view, and schedules the next tick. Past the end, it clears highlights and sets status to done.

Why save the originals? The extension mutates the page's real DOM. Without a restore path the page would be permanently altered — reader.originals is the undo log, and any future DOM code must keep it intact.
End to end

Putting it together: a live run

1

You press Command+Shift+Y. Chrome wakes background.js and fires commands.onCommand with "toggle-reader".

2

background.js finds the active tab and calls ensureContentScript(). The ping fails (first time), so it injects content.js into the page.

3

content.js boots, sets the __pageSpotlightSpeedReader guard flag, and registers its message listener. background.js then sends SPEED_READER_TOGGLE.

4

content.js finds the readable root, wraps words in spans, injects the pssr-* styles, highlights the first chunk, and starts the setTimeout pacing loop. Status: running.

5

You open the popup. popup.js pings (succeeds this time), asks SPEED_READER_STATUS, and renders "Running" plus the word / time estimate.

6

You press Command+Shift+Up. background.js bumps WPM by 25 in chrome.storage.sync and sends SPEED_READER_SETTINGS. content.js reschedules; the open popup updates live via storage.onChanged.

7

You click Stop. SPEED_READER_STOP triggers reset(): the timer clears, the HUD is removed, and every wrapper span is swapped back for its original text node — the page is untouched again.

Why constants are duplicated

DEFAULT_SETTINGS (wpm 350, chunkSize 1, autoScroll true), clamp(), isSupportedUrl(), and the clamp ranges appear in all three of background.js, popup.js, and content.js. The three contexts can't share JavaScript modules, so each carries its own copy — keep them in sync when changing values.
