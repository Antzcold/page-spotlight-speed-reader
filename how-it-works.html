How the Page Spotlight Speed Reader Works

A tour of the five files that make the Chrome extension tick

This extension is a Manifest V3 Chrome extension written in plain JavaScript — no build step, no frameworks. The five files live in three isolated execution contexts that can only talk to each other by passing messages. Understanding that separation is the key to understanding everything else.

The three worlds

Chrome deliberately keeps extension code in separate sandboxes. A background worker cannot touch a web page's DOM; a popup cannot run forever; a content script cannot reach the popup's buttons. So the files split responsibilities like this:

manifest.json manifest

The config file Chrome reads first. It declares the extension's name, permissions, the background service worker, the popup, and the keyboard shortcuts. Nothing here runs logic — it just wires the other pieces together.

background.js background worker

An invisible, event-driven service worker. It wakes up when a keyboard shortcut is pressed, figures out which tab is active, makes sure the content script is loaded in that tab, and forwards a command. It owns no UI.

popup.html / popup.js popup

The little settings panel that appears when you click the extension icon. It shows WPM, chunk size, auto-scroll, status, and a word/time estimate. It opens and closes with each click — it is not persistent.

content.js content script

The muscle. Once injected, it runs inside the web page. It finds the article, wraps each word in a span, and highlights one chunk at a time on a timer. It is the only file that touches the page's DOM.

1. manifest.json — the wiring diagram

Everything starts here. Chrome reads this before any code runs. The important parts: 

"permissions": ["activeTab", "scripting", "storage", "tabs"],
"host_permissions": ["<all_urls>"],
"background": { "service_worker": "background.js" },
"action": { "default_popup": "popup.html" },
"commands": {
  "toggle-reader": { "suggested_key": { "default": "Alt+Y", "mac": "Command+Shift+Y" } },
  "increase-speed": { ... },
  "decrease-speed": { ... },
  "stop-reader":    { ... }
}

Two permissions matter most: scripting lets the extension inject content.js on demand (note: content.js is not listed under content_scripts — it is loaded lazily), and storage lets settings persist across sessions. The commands block maps keyboard shortcuts to named actions that the background worker listens for.

Why lazy injection? Listing a content script in the manifest would force it into every page automatically, costing memory and risking breakage. Instead, content.js is loaded only when the user actually invokes the reader on a specific tab.

2. background.js — the keyboard dispatcher

The worker's only entry point is a single listener:

chrome.commands.onCommand.addListener(async (command) => {
  const [tab] = await chrome.tabs.query({ active: true, currentWindow: true });
  if (!tab?.id || !isSupportedUrl(tab.url)) return;
  await ensureContentScript(tab.id);
  // ...then send the right SPEED_READER_* message for this command
});

The flow for every shortcut is identical:

  1. Find the active tab; bail out if it's not http(s).
  2. Call ensureContentScript() — a ping-then-inject pattern.
  3. Send a message telling content.js what to do.
async function ensureContentScript(tabId) {
  try {
    await chrome.tabs.sendMessage(tabId, { type: "SPEED_READER_PING" });
  } catch {
    await chrome.scripting.executeScript({ target: { tabId }, files: ["content.js"] });
  }
}

It pings the tab. If the ping succeeds, content.js is already there. If it throws, the script isn't loaded yet, so it injects it. This makes the shortcut idempotent — press it on a fresh tab and it loads + starts; on a tab already running it just toggles.

For speed up/down it also updates chrome.storage.sync and sends a SPEED_READER_SETTINGS message. Writing to storage matters because the popup listens for storage changes — so a shortcut-pressed speed change shows up live in an open popup.

3. popup.html + popup.js — the control panel

The HTML is just markup: status text, Start/Stop/Reset buttons, a WPM stepper, chunk size, and an auto-scroll checkbox. popup.js gives it behavior. On open, init() runs: 

const settings = await chrome.storage.sync.get(DEFAULT_SETTINGS);
applySettingsToInputs(settings);
const [tab] = await chrome.tabs.query({ active: true, currentWindow: true });
if (!state.supported) { renderStatus("unsupported"); return; }
await ensureContentScript();
const response = await sendMessage({ type: "SPEED_READER_STATUS" });
renderStatus(response?.status ?? "ready");
await refreshEstimate();

The popup reuses the same ping-then-inject pattern as the background worker, then asks content.js for its current status and a word count so it can show "Article words / Est. time" immediately.

Buttons call sendMessage with the appropriate SPEED_READER_* type. The popup reads back the { status, estimate } response and updates its labels. Crucially, it also registers a storage listener: 

chrome.storage.onChanged.addListener((changes, areaName) => {
  if (areaName !== "sync") return;
  // merge changed settings into the inputs and recompute the time estimate
});

This is what makes a keyboard-shortcut speed change (handled by the background worker) reflect instantly in a popup that happens to be open.

4. content.js — the reading engine

This is where the actual work happens. A single IIFE guards against double-injection with a window flag: 

if (win.__pageSpotlightSpeedReader) return;
// ...set up everything...
win.__pageSpotlightSpeedReader = true;

Because Chrome can inject content.js more than once, this flag prevents two readers from running on the same page.

The message handler

All communication from background/popup arrives at one listener. Each message type maps to an action:

Message type What content.js does
SPEED_READER_PING Returns status only — used to detect whether the script is loaded.
SPEED_READER_STATUS / _ESTIMATE Returns current status + word count, without changing anything.
SPEED_READER_TOGGLE Applies settings, then start / pause / resume depending on current state.
SPEED_READER_STOP Stops and tears down (restores original DOM).
SPEED_READER_RESET Restores the page to its original state.
SPEED_READER_SETTINGS Updates WPM/chunk/scroll; if running, re-highlights and reschedules.

Every response is the same shape — { status, estimate? } — so the popup can render any reply generically.

Finding the article

findReadableRoot() scores candidate elements (article, main, section, div) by visible text length plus 250 points per substantial <p>, with a heading bonus. It prefers a direct article/main/[role='main'] that scores ≥ 120, otherwise picks the highest-scoring candidate ≥ 240. If the chosen root covers less than 85% of the page's readable words, it expands to the common ancestor of all readable blocks. Excluded selectors (nav, header, footer, ads, forms) and "distraction" heuristics (class/id matching ad, newsletter, subscribe…) filter out junk.

Wrapping words

wrapWords() walks the readable blocks (p, li, blockquote, h1–h3, plus leaf divs on pages like social apps that lack real <p>s), splits each text node into words/whitespace, and replaces the text with <span class="pssr-word"> elements. The original text node is saved in reader.originals so reset() can swap it back and leave the page exactly as it found it.

Pacing

scheduleNext() chains setTimeout calls. The interval is (60000 / wpm) * chunkSize, floored at 80ms so very high WPM can't freeze the page. Each tick advances the index by chunkSize, applies the pssr-active class to the current chunk (and pssr-read to everything before it), optionally smooth-scrolls the active word into view, and schedules the next tick. When the index passes the end, it clears highlights and sets status to done.

Why save the originals? The extension mutates the page's real DOM (wrapping every word in spans). Without a restore path, the page would be permanently altered. reader.originals is the undo log — any future DOM code must keep it intact.

Putting it all together: a live run

1
You press Command+Shift+Y. Chrome wakes background.js and fires commands.onCommand with "toggle-reader".
2
background.js finds the active tab, calls ensureContentScript(). The ping fails (first time), so it injects content.js into the page.
3
content.js boots, sets the __pageSpotlightSpeedReader guard flag, and registers its message listener. background.js then sends SPEED_READER_TOGGLE.
4
content.js finds the readable root, wraps words in spans, injects the pssr-* styles, highlights the first chunk, and starts the setTimeout pacing loop. Status becomes running.
5
You open the popup. popup.js pings (succeeds this time), asks SPEED_READER_STATUS, and renders "Running" plus the word/time estimate.
6
You press Command+Shift+Up. background.js bumps WPM by 25 in chrome.storage.sync and sends SPEED_READER_SETTINGS. content.js reschedules the loop; the open popup's storage.onChanged listener updates the WPM field live.
7
You click Stop (or press Command+Shift+S). A SPEED_READER_STOP message triggers reset(), which clears the timer, removes the HUD, and swaps each wrapper span back for its original text node — the page is restored to its untouched state.

Why constants are duplicated

DEFAULT_SETTINGS (wpm 350, chunkSize 1, autoScroll true), clamp(), isSupportedUrl(), and the clamp ranges appear in background.js, popup.js, and content.js. The three contexts can't share JavaScript modules, so each carries its own copy. Keep them in sync when changing values.
